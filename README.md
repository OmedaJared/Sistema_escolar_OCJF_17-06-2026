Documentación del Sistema Escolar (Flask + MongoDB)

1. Arquitectura y Tecnologías UtilizadasLa aplicación está desarrollada bajo el patrón de arquitectura monolítica y utiliza el framework ágil Flask en conjunto con la base de datos NoSQL MongoDB.ComponenteTecnologíaPropósitoBackend FrameworkFlaskEnrutamiento, procesamiento de peticiones HTTP y renderizado de plantillas.Base de DatosMongoDBAlmacenamiento de datos en formato de documentos BSON (JSON binario).ODM / Driver de BDPyMongoCliente oficial de MongoDB para interactuar con la base de datos desde Python.Variables de EntornoPython-dotenvGestión y aislamiento de credenciales y datos de configuración del sistema.
 
 2. Librerías y Dependencias del SistemaA continuación se detallan los módulos importados en el proyecto y su función específica dentro de la aplicación:flask:Flask: Clase principal para instanciar la aplicación web.render_template: Función para renderizar y pasar variables a los archivos HTML (usando el motor Jinja2).request: Objeto global para capturar los datos enviados por el usuario mediante formularios (GET y POST).redirect y url_for: Herramientas para la navegación interna; redirigen el flujo del navegador hacia otras rutas del sistema de forma dinámica.flash: Sistema de mensajería emergente temporal (Flash Messages) utilizado para notificar éxitos, advertencias o errores al usuario en la interfaz.pymongo.MongoClient: Driver oficial que permite abrir y gestionar el pool de conexiones hacia el servidor local o remoto de MongoDB.bson.objectid.ObjectId: Clase indispensable para manejar el identificador único (_id) de 12 bytes generado automáticamente por MongoDB, permitiendo realizar búsquedas, ediciones y eliminaciones precisas.dotenv.load_dotenv: Función que carga las variables de entorno guardadas en un archivo físico .env al sistema operativo del servidor, ideal para proteger cadenas de conexión y nombres de bases de datos.os: Módulo estándar de Python utilizado para interactuar con las variables del sistema cargadas previamente por dotenv.datetime: Librería nativa para el futuro control, formateo o manipulación de fechas en los registros escolares.
 
 3. Configuración de la Base de Datos (database.py)Este archivo centraliza el acceso a MongoDB, previene la duplicidad de conexiones y asegura la integridad de los datos mediante el uso de índices únicos a nivel de software/base de datos.Variables de Configuración:MONGO_URI: Cadena de conexión. Si no se encuentra en las variables de entorno, por defecto apunta al servidor local: mongodb://localhost:27017/.DB_NAME: Nombre de la base de datos a utilizar (por defecto sistema_escolar_db).Inicialización e Índices Únicos:El script ejecuta un bloque try-except al arrancar. Si la conexión es exitosa, genera las siguientes restricciones de unicidad en las colecciones correspondientes:db.alumnos.create_index("matricula", unique=True) $\rightarrow$ Impide que dos alumnos tengan el mismo número de matrícula.db.maestros.create_index("num_empleado", unique=True) $\rightarrow$ Asegura que el número de empleado de los docentes sea irrepetible.db.materias.create_index("codigo_materia", unique=True) $\rightarrow$ Garantiza códigos de asignaturas únicos en el plan de estudios.Función Exportable:get_db(): Retorna la instancia activa del objeto db hacia otros módulos de la aplicación (como app.py).
 
 4. Estructura de Funciones y Rutas (app.py)Nota de Conflicto de Conexión: Al inicio de app.py, el código invoca correctamente a db = get_db(). Sin embargo, inmediatamente abajo sobreescribe la variable db volviendo a conectar de forma manual a client["mi_base_de_datos"]. Esto causa que los índices únicos definidos en database.py no tengan efecto en la base de datos manual mi_base_de_datos. Se recomienda remover la conexión explícita de app.py y mantener únicamente la función get_db().A continuación, se desglosan las funciones controladoras de rutas agrupadas por módulos CRUD (Create, Read, Update, Delete):Raíz del Sistemaindex() [Ruta: / | Método: GET]Descripción: Renderiza la plantilla principal base.html que sirve como la página de bienvenida y el contenedor global de la interfaz del sistema.Módulo de Alumnosalumnos() [Ruta: /alumnos | Métodos: GET, POST]Comportamiento POST (Create): Captura los datos del formulario (matricula, nombres, apellidos, correo, fecha_nacimiento). Valida que no haya campos vacíos y verifica en la base de datos si la matrícula ya existe. Si todo es correcto, inserta el registro del alumno y emite un mensaje de éxito.Comportamiento GET (Read): Consulta todos los documentos de la colección db.alumnos, los convierte a una lista nativa de Python y los envía a la plantilla alumnos.html para su visualización.editar_alumno(id) [Ruta: /alumnos/editar/<id> | Método: POST]Descripción: Recibe el ID único del alumno en la URL, limpia los nuevos valores enviados en el formulario y, mediante el operador de actualización $set de MongoDB, modifica los datos del alumno correspondiente a ese ObjectId(id).eliminar_alumno(id) [Ruta: /alumnos/eliminar/<id> | Método: GET]Descripción: Recibe el parámetro id, busca al alumno mediante su ObjectId y ejecuta un borrado físico definitivo del documento en la colección.Módulo de Maestrosmaestros() [Ruta: /maestros | Métodos: GET, POST]Comportamiento POST (Create): Recibe parámetros laborales del docente (num_empleado, nombre_completo, especialidad, telefono, turno). Realiza comprobaciones de campos obligatorios y unicidad del número de empleado antes de guardarlo en la colección db.maestros.Comportamiento GET (Read): Extrae la lista completa de maestros y la inyecta dinámicamente en el archivo maestros.html.editar_maestro(id) [Ruta: /maestros/editar/<id> | Método: POST]Descripción: Actualiza los campos de texto del docente identificado por su ObjectId(id). Si hay campos vacíos, cancela la operación enviando una alerta tipo flash.eliminar_maestro(id) [Ruta: /maestros/eliminar/<id> | Método: GET]Descripción: Remueve de forma permanente el registro del maestro seleccionado de la colección de la base de datos.Módulo de Materias / Asignaturasmaterias() [Ruta: /materias | Métodos: GET, POST]Comportamiento POST (Create): Recibe datos de la asignatura (codigo_materia, nombre_asignatura, horas_semanales, creditos). Valida la obligatoriedad y la unicidad del código. Utiliza un bloque try-except con ValueError para forzar la conversión de las horas y los créditos a tipos de datos enteros (int) en la base de datos.Comportamiento GET (Read): Recupera los planes de estudio vigentes de la colección db.materias y renderiza la vista materias.html.editar_materia(id) [Ruta: /materias/editar/<id> | Método: POST]Descripción: Modifica los atributos de la materia seleccionada. Incluye la misma validación numérico-entera para las horas y créditos usando la estructura try-except.eliminar_materia(id) [Ruta: /materias/eliminar/<id> | Método: GET]Descripción: Suprime el documento de la materia correspondiente al ObjectId(id) indicado de la base de datos.
 
 5. Arranque del ServidorAl final de app.py, el bloque condicional if __name__ == '__main__': se encarga de ejecutar el servidor web local bajo los siguientes parámetros:debug=True: Activa el modo de depuración en tiempo real. Si modificas el código, el servidor se reinicia solo y muestra errores detallados en el navegador en caso de fallos.host='0.0.0.0': Permite que la aplicación sea accesible desde cualquier dispositivo conectado a la misma red local (LAN) y no solo desde localhost.port=5000: Expone el sistema a través del puerto de red estándar 5000.
